# ワイヤーフレーム制作ルール

このファイルはClaude Codeがワイヤーフレームを生成するときに従うルール集です。
必ずこのファイルを読んでからワイヤーフレームの生成を開始してください。

---

## 基本方針

- ワイヤーフレームはデザインではなく「構造の設計図」として作ること
- 装飾・色・フォントの指定はしない（グレースケールのみ）
- コンテンツの配置・優先順位・導線を明確にすることを最優先にする
- クライアントが「何をどこに置くか」をイメージできるレベルで作ること

---

## 対応デバイス

- PCのみ（基準幅：1440px）
- コンテンツ幅：1200px中央寄せを基本とする

---

## ページ共通ルール

### ヘッダー
- 必ず全ページに入れること
- 入れる要素：ロゴ（左寄せ）、グローバルナビ（右寄せ）、CTAボタン（右端）
- CTAボタンはhearing.mdの「してほしいアクション」に合わせること
  （例：お問い合わせ・資料請求・予約する など）

### フッター
- 必ず全ページに入れること
- 入れる要素：ロゴ、コピーライト、ナビゲーションリンク

### CTA（Call To Action）
- 各ページに必ず1箇所以上設置すること
- ページの内容に応じた自然な位置に配置すること

---

## セクション構成のルール

セクション構成は固定ではなく、hearing.mdの内容をもとにClaudeが最適な構成を考えること。
サイトの種別・目的・ターゲット・サービス内容を読み取り、そのクライアントに最適な順番と内容で構成すること。

### 必ず含めること（全ページ共通）
- ヘッダー
- ヒーローセクション（H1キャッチコピー・サブコピー・CTAボタン・メインビジュアル）
- CTAセクション（各ページ最低1箇所）
- フッター

### セクションを考えるときの判断基準
- **サイトの目的**：何を達成したいか（問い合わせ獲得・採用・販売など）
- **ターゲット**：誰に向けて何を伝えるべきか
- **サービス内容**：何を・どの順番で見せると伝わりやすいか
- **クライアントの強み**：何を前面に出すべきか
- **してほしいアクション**：どこにCTAを置くと自然か

### セクション構成を提案するときのルール
- 仮構成の提案時に各ページのセクション一覧と、各セクションの目的を一言で添えること
- 「なぜこの順番にしたか」の理由を簡潔に説明すること
- ユーザーからの修正依頼には柔軟に対応すること

---

## セクションカタログ

以下は使用できるセクションの一覧と、使う判断基準。
パターンに当てはめるのではなく、hearing.mdの内容をもとにここから必要なものを選んで組み合わせること。

| セクション名 | 目的 | 使う条件の例 |
|---|---|---|
| ヒーロー | 第一印象・キャッチコピー・CTA | 全ページ必須 |
| 課題提起 | ターゲットの悩みに共感・問題提起 | ターゲットがまだ解決策を探していない段階、自覚が薄い悩みを持つ場合 |
| サービス概要 | 何ができるかをざっくり伝える | ほぼ全ページで使用 |
| 特徴・強み | 競合との差別化ポイントを伝える | 同業他社が多い、比較検討が発生しやすいサービスの場合 |
| 実績・数字 | 定量的な信頼担保 | 実績・規模感が強みの場合、BtoB向け |
| サービス詳細・料金 | 具体的な内容と費用感を伝える | 価格が明示できる、価格透明性が信頼につながる場合 |
| 導入フロー・プロセス | 申し込みから完了までの流れを示す | 手順が複雑、初めての人が多い、心理的ハードルが高い場合 |
| お客様の声 | 第三者視点の信頼形成 | 知名度が低い、BtoCで感情訴求が重要、比較検討中のユーザーが多い場合 |
| 事例・実績詳細 | 具体的な成果・Before/Afterの提示 | BtoBで効果や成果を証明する必要がある場合 |
| よくある質問（FAQ） | 不安・疑問を先回りして解消する | 問い合わせ前の迷いが多い、よくある誤解がある場合 |
| チーム・代表挨拶 | 人への信頼・専門性の可視化 | 人柄や専門性が選ばれる理由になる（士業・コンサル・個人事業主など） |
| メディア掲載・受賞 | 外部からの権威付け | 掲載・受賞実績がある場合 |
| アクセス・MAP | 来店・来社への誘導 | 実店舗・オフィスへの来訪が発生する場合 |
| CTA（中間・末尾） | 行動喚起 | 各ページ必ず1箇所以上。長いページはスクロール途中にも設置 |

## 構成設計の推論フレーム

セクション構成を提案する前に、以下の問いに順番に答えてから組み立てること。
**この推論プロセスを飛ばして構成を決めないこと。**

---

### PHASE 1：ターゲットの状態を把握する

まずユーザーがどういう状態でサイトに来るかを把握する。
ここを間違えると、どんなにセクションを並べても刺さらないサイトになる。

**確認する項目：**
- このサービスをまだ知らない人が来る？すでに知っていて比較検討中？
- ターゲットは「問題を自覚している」のか、「まだ気づいていない」のか
- BtoC（個人・感情で動く）か、BtoB（法人・複数の意思決定者が関わる）か
- サイトに来る前にどこで何を見てきた人か（検索・SNS・紹介など）

---

### PHASE 2：サイト全体の動線を設計する

次に「サイト全体でユーザーをどう動かすか」を設計する。
各ページの役割を決めてから、ページ間のつながりを設計する。

**各ページの役割を決める：**

| ページの役割 | 内容 |
|---|---|
| 興味喚起 | 「このサービスは自分に関係ある」と思わせる（主にTOP） |
| 信頼形成 | 「この会社は信頼できる」と思わせる（ABOUT・実績など） |
| 説得 | 「このサービスを使いたい」と思わせる（SERVICE・料金など） |
| 背中を押す | 「今すぐ問い合わせよう」と思わせる（CONTACT・CTA） |

**ページ間の導線を設計する：**
- TOPから各ページへの誘導：どのセクションからどのページに飛ばすか
- 各ページからCONTACTへの誘導：どこで背中を押すか
- 迷ったユーザーの受け皿：FAQやお客様の声でフォローする導線があるか
- 離脱防止：ユーザーが迷子になったとき戻れる導線があるか（パンくず・関連ページリンクなど）

**ページ間の導線を提案するときのルール：**
- 各ページのCTAがどこに向いているかを明記すること
- 「ページAのCTAセクション → ページBへ誘導」のような形で整理すること

---

### PHASE 3：コンバージョンまでの障壁を特定する

最終アクション（問い合わせ・購入・来店など）に至るまでの障壁を特定する。
障壁を取り除くセクションを意図的に配置することが重要。

**確認する項目：**
- 最終アクションの心理的ハードルはどのくらいか（高い・普通・低い）
- ターゲットが行動をためらう理由は何か
  - 価格が高い・わからない → 料金の明示・費用対効果の説明
  - 信頼できるか不安 → 実績・お客様の声・代表メッセージ
  - 手順がわからない → 導入フロー・FAQ
  - 本当に自分に合うか不安 → 事例・ターゲット明示・FAQ

---

### PHASE 4：信頼獲得の方法を決める

何でどうやって信頼を得るかを決める。
クライアントの状況によって「何で信頼を得るか」は変わる。

| 信頼の源泉 | 使う場面 |
|---|---|
| 実績・数字 | 規模感が強み、BtoB向け |
| 人柄・ストーリー | 個人・小規模、感情訴求が重要なBtoC |
| 第三者の声 | 知名度が低い、比較検討が発生しやすい |
| 専門性・資格 | 士業・コンサル・医療など |
| メディア掲載・受賞 | 外部からの権威付けが有効な場合 |

---

### PHASE 5：セクションを選んで順番を決める

PHASE 1〜4の答えをもとに、セクションカタログから必要なものを選んで順番を決める。

**守ること：**
- 「なぜこの順番にしたか」を一言で説明できるように組み立てること
- ページをまたいだ導線（どこからどこへ誘導するか）も合わせて提示すること
- ユーザーに提案するときはPHASE 1〜4の判断根拠も簡潔に添えること

---

## Figmaへの出力ルール

- 1ページにつき1フレームを作成すること
- フレーム名は「ページ名_PC」とすること（例：TOP_PC、ABOUT_PC）
- 各セクションにセクション名をテキストで入れること
- 画像が入る箇所は必ずグレーの矩形でプレースホルダーを表示すること
- テキストはダミーテキストではなく、hearing.mdの内容をもとにした仮テキストを入れること

---

## 禁止事項

- hearing.mdに記載のない情報を勝手に追加しないこと
- デザイン（色・装飾）の指定をしないこと
- ページ構成の確認を取らずにFigma出力を開始しないこと

---

## テキストの文字量基準

ワイヤーフレームに入れる仮テキストは以下の文字量を守ること。
ダミーテキスト（「テキストが入ります」など）は使用禁止。
必ずhearing.mdの内容をもとにした意味のある仮テキストを入れること。

| テキストの種類 | 文字量の目安 |
|---|---|
| 大見出し（H1・キャッチコピー） | 15〜25文字 |
| 中見出し（H2・セクションタイトル） | 10〜20文字 |
| 小見出し（H3・カード見出しなど） | 8〜15文字 |
| 本文・説明文 | 60〜120文字（3〜5行相当） |
| リスト項目 | 1項目につき30〜60文字 |
| ボタンテキスト | 8〜15文字 |
| ナビゲーションメニュー | 1項目につき4〜8文字 |

---

## テキストを作るタイミング

テキストは以下の2段階で作成すること。

### ステップ1：仮構成の提案時（Figma出力前）

ページ構成をテキストで提案するときに、以下もあわせて提示すること。

- 各ページの大見出し（H1）の仮テキスト案
- トップページの各セクションの中見出し（H2）の仮テキスト案
- ヒーローセクションのキャッチコピーとサブコピーの仮テキスト案
- CTAボタンのテキスト案

ユーザーに確認を取り、修正があれば反映してからステップ2に進むこと。

### ステップ2：Figma出力時

ステップ1で確認済みのテキストをベースに、全セクションの仮テキストを作成してFigmaに入れること。
本文・説明文など、ステップ1で省略した箇所もすべて埋めること。

---

## Figmaの出力構成ルール

### 全ページを1つのFigmaページに横並びで出力すること

- Figmaのページ（左サイドバーのページ）は **1つだけ** 作成すること
- ページ名は `ワイヤーフレーム` とすること
- 全ページのフレームを **横1列に並べて** 配置すること
- フレーム間の余白は **200px** とすること
- 左から順番に以下の順で並べること
  1. TOP_PC
  2. ABOUT_PC
  3. SERVICE_PC
  4. NEWS_PC（ある場合）
  5. NEWS_DETAIL_PC（NEWS_PCがある場合は必ずセットで作成）
  6. CONTACT_PC
  7. 404_PC（デフォルトで必ず作成）

### お知らせページのルール

- お知らせ一覧ページ（NEWS_PC）を作成する場合は、**お知らせ詳細ページ（NEWS_DETAIL_PC）も必ずセットで作成すること**
- NEWS_DETAIL_PCの構成：ヘッダー／パンくずリスト／記事タイトル・アイキャッチ画像・日付・カテゴリ／記事本文エリア（h2〜h4、リスト、引用、リンク、本文テキストをテスト投稿のように記載）／一覧に戻るボタン／関連記事／CTA／フッター

### 404ページのルール

- 404ページ（404_PC）は**ヒアリング内容に関わらず、デフォルトで必ず作成すること**
- 404_PCの構成：ヘッダー／404メッセージ（「ページが見つかりません」等）／トップへ戻るボタン／フッター

### 理由

1ページにまとめることで、クライアントへの共有・確認がしやすくなる。
ページを分けるとFigmaの左サイドバーを切り替える手間が増えるため、
WF確認フェーズでは横スクロールで全体を見渡せる構成が望ましい。
